## CSF Overview

The Command Sequence File (CSF) is the artifact that instructs HABv4 how to authenticate a boot image. It exists in two forms:

1. **CSF source file** (`.csf`): Human-readable text file, input to the `cst` tool
2. **CSF binary** (`.bin`): Compiled binary embedded in the flash image alongside the IVT

The CSF binary is self-contained: it includes the command structure, key certificates, and CMS signature blobs all in a single contiguous binary region. The IVT `csf` field points to the start of this binary in memory.

---

## CSF Binary Format

The CSF binary is a sequence of tagged structures, similar in concept to ASN.1 TLV encoding but with NXP-specific tags:

```
CSF Binary Layout:
┌────────────────────────────────────────────────────────┐
│  CSF Header (4 bytes)                                  │
│  tag=0xD4, length=total_csf_size (BE), version=0x43    │
├────────────────────────────────────────────────────────┤
│  Command 1: Install SRK                                │
│  tag=0xC0, length, params                              │
│  [embedded SRK table binary data]                      │
├────────────────────────────────────────────────────────┤
│  Command 2: Install CSFK                               │
│  tag=0xBF, length, params                              │
│  [CSF key certificate DER data]                        │
├────────────────────────────────────────────────────────┤
│  Command 3: Authenticate CSF                           │
│  tag=0xBF, length, params                              │
│  [CMS SignedData blob]                                 │
├────────────────────────────────────────────────────────┤
│  Command 4: Install Key (IMG key)                      │
│  tag=0xBF, length, params                              │
│  [IMG key certificate DER data]                        │
├────────────────────────────────────────────────────────┤
│  Command 5: Authenticate Data                          │
│  tag=0xCA, length, params                              │
│  [CMS SignedData blob with image hash/signature]       │
└────────────────────────────────────────────────────────┘
```

All multi-byte numeric fields in the CSF binary are **big-endian** unless otherwise noted.

### CSF Command Structure

Each command begins with a 4-byte command header:

```
Byte 0:   tag      (command identifier)
Byte 1-2: length   (big-endian, includes the 4-byte header)
Byte 3:   param    (command-specific parameter byte)
```

## CSF Commands: Detailed Reference

### Header Command

The CSF "Header" is not a command per se but the CSF file header. In the CSF source file, it is specified as:

```ini
[Header]
    Version = 4.3
    Hash Algorithm = sha256
    Engine = CAAM
    Engine Configuration = 0
    Certificate Format = X509
    Signature Format = CMS
```

The binary representation is the 4-byte CSF header at offset 0 of the CSF binary:
- `tag = 0xD4`
- `length` = total CSF binary length (big-endian 16-bit)
- `version = 0x43` for HABv4.3

**Engine options:**

| Engine Name | Code | Used For |
|-------------|------|----------|
| CAAM | 0x55 | Primary on i.MX8MP; supports RSA-2048/4096, SHA-256/384/512 |
| DCP | 0x1B | i.MX6UL/ULL only; supports RSA-2048, SHA-256 only |
| SW | 0xFF | Software fallback; always available but slow |
| SAHARA | 0x06 | Legacy (i.MX53) |

**Engine Configuration:**

```
0x00 = Default configuration
For CAAM:
  Bits [3:0] = AES key size selection (0=default)
  Bits [7:4] = PKHA precision (0=default = 2048-bit RSA)
```

For SHA-384/SHA-512 or RSA-4096, `Engine Configuration` must be set to the appropriate value. For standard RSA-2048/SHA-256, `0` is correct.

### Install SRK Command

Source file syntax:
```ini
[Install SRK]
    File = "crts/SRK_1_2_3_4_table.bin"
    Source index = 0
```

Binary command tag: `0xC0`

Parameters:
- `File`: Path to the SRK table binary (generated by `srktool`)
- `Source index`: Which SRK entry to use (0-3); must match the key used for signing

The SRK table binary is a structured binary (not PEM) produced by `srktool`. It contains all four SRK public keys even if you only have one SRK key pair. Keys for unused slots can be placeholder entries.

**Why 4 SRKs?** The 4-SRK design enables key rotation without device bricks:
- Deploy devices with SRK 0 active
- Generate SRK 1 as backup
- If SRK 0 is compromised, burn `SRK_REVOKE[0]` fuse on all devices
- Push firmware signed with SRK 1
- Devices now require SRK 1 signature

The SRK table hash covers all 4 entries. If you change any SRK entry, the hash changes, and existing devices (with the old hash burned in fuses) will reject authentication.

---

### Install CSFK Command

Source file syntax:
```ini
[Install CSFK]
    File = "crts/CSF1_1_sha256_2048_65537_v3_usr_crt.pem"
```

Binary command tag: `0xBF` with `key_idx = 1` parameter

The CSFK certificate (`CSF1_1_...`) naming convention from CST:
- `CSF1` = CSF key type
- `1` = SRK index (which SRK signed this CSFK)
- `sha256` = hash algorithm
- `2048` = RSA key size
- `65537` = public exponent
- `v3_usr` = X.509v3 user certificate
- `crt` = certificate

---

### Authenticate CSF Command

Source file syntax:
```ini
[Authenticate CSF]
```

No parameters required. The `cst` tool automatically:
1. Computes the hash of the CSF binary up to this command
2. Signs it with the CSFK private key
3. Creates a CMS SignedData blob
4. Embeds the blob as the data payload of this command

The Authenticate CSF command signature covers:
- The CSF Header
- The Install SRK command
- The Install CSFK command
- The beginning of the Authenticate CSF command (up to the signature location)

---

### Install Key Command

Source file syntax:
```ini
[Install Key]
    Verification index = 0
    Target index = 2
    File = "crts/IMG1_1_sha256_2048_65537_v3_usr_crt.pem"
```

Binary command tag: `0xBF` with key slot parameters

Parameters:
- `Verification index = 0`: Use the SRK at slot 0 to verify (wait — the CSFK should verify the IMG key, not the SRK directly — in practice, this field refers to which HAB internal key slot to use for verification; CSFK is in slot 1, IMG goes to slot 2)
- `Target index = 2`: Store the verified IMG key in HAB internal slot 2
- `File`: IMG key certificate, signed by CSFK

> **📝 NOTE:** The `Verification index` in Install Key refers to the HAB internal key slot indices, not directly to the SRK table indices. HAB internal slots are assigned as: slot 0 = unused/SRK, slot 1 = CSFK, slot 2 = IMG key. Setting `Verification index = 0` tells HAB to use the SRK (or the key in HAB slot 0) to verify, but in practice the CSFK is in slot 1 and the Authenticate CSF step already validated the CSF. The exact semantics depend on HABv4 version; follow the NXP CST examples.

### Authenticate Data Command

Source file syntax:
```ini
[Authenticate Data]
    Verification index = 2
    Blocks = 0x7E1000 0x000 0xC000 "flash.bin"
```

Binary command tag: `0xCA`

Parameters:
- `Verification index = 2`: Use the key in HAB internal slot 2 (IMG key) for verification
- `Blocks`: One or more data blocks in format: `<memory_address> <file_offset> <length> "<filename>"`

**Block specification explained:**
```
0x7E1000  = RAM address where the data will be loaded at authentication time
0x000     = Byte offset within the named file where this data starts
0xC000    = Number of bytes to authenticate (48 KB in this example)
"flash.bin" = File containing the data (used only at signing time by cst)
```

**Critical:** The memory address `0x7E1000` is the address where the image is in RAM when HABv4 authenticates it. If the IVT says the image loads to `0x7E1000`, this must match. If they differ, authentication fails.

**Multiple block specification (for images with non-contiguous regions):**
```ini
[Authenticate Data]
    Verification index = 2
    Blocks = 0x7E1000 0x000 0x2000 "flash.bin", \
             0x7E5000 0x4000 0x8000 "flash.bin"
```

The hash is computed over the concatenation of the block data in order.

**Address calculation for i.MX8MP SPL:**

The i.MX8MP SPL is a combined flash.bin that contains:
- SPL binary
- DDR training firmware
- TF-A BL31
- OP-TEE
- U-Boot proper

The flash.bin structure built by imx-mkimage:
```
flash.bin layout (eMMC, offset from 0x8000):
Offset 0x0000: IVT
Offset 0x0020: Boot Data
Offset 0x0040: DCD (optional)
Offset 0x1000: Image data start
  ...
Offset (csf_offset): CSF

Image memory layout when loaded to RAM:
OCRAM_BASE + 0x0000: IVT (at exactly load_address)
OCRAM_BASE + 0x0020: Boot Data
...
```

To compute the correct `Blocks` address:
```
block_ram_addr = IVT_load_address  (where ROM puts the image)
block_file_offset = 0              (start of flash.bin)
block_length = csf_file_offset     (bytes before the CSF itself)

Example for i.MX8MP SPL at OCRAM:
  IVT load addr = 0x920000 (OCRAM)
  flash.bin size before CSF = 0x30000 (192 KB)
  
Blocks = 0x920000 0x000 0x30000 "flash.bin"
```

> **⚠️ WARNING:** Getting the Authenticate Data block address wrong is the #1 cause of HABv4 authentication failures. Double-check by comparing the IVT `self` field with the address in the Blocks line.

### Set Engine Command

Source file syntax:
```ini
[Set Engine]
    Engine = CAAM
    Features = MID
    UID = 0x1122334455667788
```

Binary command tag: `0xD0`

The Set Engine command is optional. It selects which hardware engine to use for subsequent cryptographic operations and can optionally bind the signature to a specific device UID. The `Features` parameter:

| Feature | Description |
|---------|-------------|
| MID | Include Manufacture ID in signature context; signature only valid for specific device |
| CLR | Clear/reset engine state |

MID binding is used for device-unique images — the signed image is only valid for a device with that specific UID. Not commonly used for general production images.

---

### Init Command

Source file syntax:
```ini
[Init]
    Engine = CAAM
```

Binary command tag: `0xB4`

Initializes the specified engine. Usually not needed when `Engine = CAAM` is specified in the Header; the Header engine specification is sufficient. Useful for multi-engine scenarios or explicit re-initialization.

---

### Unlock Command

Source file syntax:
```ini
[Unlock]
    Engine = CAAM
    Features = MID, RNG, MFG
```

Binary command tag: `0xB2`

The Unlock command disables certain security restrictions on hardware engines **after** authentication completes. This is used to keep hardware security features available for the authenticated software.

| Feature | Effect |
|---------|--------|
| RNG | Keep CAAM RNG instantiated/unlocked after HAB exit |
| MID | Unlock Manufacturing ID access |
| MFG | Manufacturing unlock (enables some debug features) |
| CAAM_MID | CAAM Manufacturing ID specific |

For production i.MX8MP:
```ini
[Unlock]
    Engine = CAAM
    Features = RNG
```
This keeps the CAAM RNG available to U-Boot and Linux after HABv4 completes. Without this, U-Boot must re-instantiate the RNG itself.

---

### Encrypt Data Command (Optional Confidentiality)

Source file syntax:
```ini
[Install Secret Key]
    Verification index = 0
    Target index = 3
    Key = "aes_key.bin"
    Key identifier = 0x12345678
    Blob address = 0x7E2000

[Decrypt Data]
    Verification index = 3
    Mac Bytes = 16
    Blocks = 0x7E3000 0x0 0x8000 "encrypted.bin"
```

Encryption is optional and not required for integrity-only Secure Boot. When used:
- The AES-256 image encryption key is wrapped in a CAAM key blob
- The blob is embedded in the CSF
- CAAM decrypts the blob using its device-unique Master Key (derived from fuses)
- The decrypted AES key is used to decrypt the image data before authentication
- This provides both confidentiality and integrity

> **📝 NOTE:** Encryption in HABv4 requires the CAAM Master Key (derived from SRK/fuse material) to be properly set up. Encrypted images are device-specific: they can only be decrypted on devices with the same CAAM Master Key. This makes image distribution more complex. Most production deployments use integrity-only (signed but not encrypted) HABv4.

## Complete CSF for i.MX8MP: SPL + U-Boot Combined Image

This is a complete, production-ready CSF source file for signing the i.MX8MP `flash.bin` (SPL + TF-A + OP-TEE + U-Boot combined image):

```ini
# CSF for i.MX8MP flash.bin (imx-boot)
# Signs the entire boot container for eMMC boot
#
# Tested with:
#   CST 3.3.1
#   imx-mkimage lf-6.1.55-2.2.0
#   Target board: phyboard-pollux-imx8mp

[Header]
    Version = 4.3
    Hash Algorithm = sha256
    Engine = CAAM
    Engine Configuration = 0
    Certificate Format = X509
    Signature Format = CMS

[Install SRK]
    File = "crts/SRK_1_2_3_4_table.bin"
    Source index = 0

[Install CSFK]
    File = "crts/CSF1_1_sha256_2048_65537_v3_usr_crt.pem"

[Authenticate CSF]

[Install Key]
    Verification index = 0
    Target index = 2
    File = "crts/IMG1_1_sha256_2048_65537_v3_usr_crt.pem"

[Authenticate Data]
    Verification index = 2
    Blocks = 0x920000 0x000 0x2E000 "flash.bin"

[Unlock]
    Engine = CAAM
    Features = RNG
```

Where:
- `0x920000` = SPL load address in OCRAM (from IVT `self` and `boot_data.start`)
- `0x000` = data starts at offset 0 in flash.bin
- `0x2E000` = 184 KB, the size of the authenticated region (must cover all image data, up to but not including the CSF offset)
- The CSF itself is appended AFTER offset 0x2E000 in flash.bin

**Important:** The `0x2E000` length must be computed from the actual flash.bin layout. Use the imx-mkimage log output which prints the CSF offset:

```bash
# After running imx-mkimage:
# Output example:
# spl_fd_size = 0x2e000
# CSF Offset = 0x2e000
```

## Address Calculation Workflow

For any image, follow this procedure to compute the correct `Blocks` parameters:

### Step 1: Build the unsigned image

```bash
cd imx-mkimage
make SOC=iMX8MP flash_evk_flexspi   # for SPI-NOR
# or
make SOC=iMX8MP flash_evk            # for eMMC
```

The Makefile outputs something like:
```
Generated flash.bin
IVT offset in flash: 0x8000
SPL load address: 0x920000
SPL entry point: 0x920000
CSF offset in flash.bin: 0x2E000
```

### Step 2: Extract IVT fields with hexdump

```bash
# Dump the IVT (32 bytes at offset 0 of flash.bin)
hexdump -C -n 32 flash.bin
```

Expected output:
```
00000000  d1 00 20 43 00 00 92 00  00 00 00 00 00 00 00 00  |.. C............|
00000010  20 a9 92 00 00 00 92 00  00 00 00 00 00 00 00 00  | ...............|
```

Parsing:
- Byte 0: `0xd1` = IVT tag ✓
- Byte 1-2: `0x0020` = 32 bytes IVT length ✓
- Byte 3: `0x43` = HABv4.3 ✓
- Bytes 4-7: `0x00920000` = entry point (little-endian)
- Bytes 20-23: `0x00920000` = IVT self address (= load address)
- Bytes 24-27: `0x00000000` = CSF pointer (0 = not yet set, filled after signing)

### Step 3: Determine block parameters

```
RAM address  = IVT.self = 0x920000
File offset  = 0x000 (start of flash.bin)
Length       = CSF_offset - 0 = 0x2E000

Blocks = 0x920000 0x000 0x2E000 "flash.bin"
```

### Step 4: Update the CSF and sign

```bash
# Edit csf_spl.csf with correct Blocks address
# Then run:
./cst -i csf_spl.csf -o csf_spl.bin

# Inject CSF into flash.bin at CSF offset
dd if=csf_spl.bin of=flash.bin bs=1 seek=$((0x2E000)) conv=notrunc
```

### Step 5: Verify the signed flash.bin

```bash
# Confirm IVT CSF pointer is now set:
hexdump -C -s 24 -n 4 flash.bin
# Should show non-zero CSF pointer
```

## CSF for SPL-Only Signing (Non-combined)

Some workflows sign only the SPL, allowing U-Boot to be loaded separately. In this case:

```ini
[Header]
    Version = 4.3
    Hash Algorithm = sha256
    Engine = CAAM
    Engine Configuration = 0
    Certificate Format = X509
    Signature Format = CMS

[Install SRK]
    File = "crts/SRK_1_2_3_4_table.bin"
    Source index = 0

[Install CSFK]
    File = "crts/CSF1_1_sha256_2048_65537_v3_usr_crt.pem"

[Authenticate CSF]

[Install Key]
    Verification index = 0
    Target index = 2
    File = "crts/IMG1_1_sha256_2048_65537_v3_usr_crt.pem"

[Authenticate Data]
    Verification index = 2
    Blocks = 0x920000 0x000 0x20000 "u-boot-spl.bin"
```

Here the address `0x920000` and length `0x20000` are SPL-specific.

## CST Command Execution

The CST tool compiles the CSF source into the CSF binary:

```bash
# Basic usage:
./cst --i input.csf --o output.csf.bin

# With verbose output:
./cst --i input.csf --o output.csf.bin --verbose

# Verify an existing CSF binary:
./cst --verify output.csf.bin
```

CST directory layout after installation:
```
cst-3.3.1/
├── linux64/
│   ├── bin/
│   │   ├── cst          # Main CST tool (CSF compiler + signer)
│   │   └── srktool      # SRK table generator
│   └── lib/             # Required shared libraries
├── crts/                # Certificate directory (populate with your keys)
├── keys/                # Private key directory (populate with your keys)
├── docs/
│   ├── CST_UM.pdf       # CST User Manual
│   └── HAB4_API.pdf     # HABv4 API Reference
└── scripts/
    └── hab4_pki_tree.sh # PKI tree generation script
```

After running CST:
```bash
# Successful execution output:
CSF Loaded successfully!!
```

If there are errors:
```bash
# Example error:
Error: failed to open key file crts/IMG1_1_sha256_2048_65537_v3_usr_crt.pem
# → Check paths relative to the CSF file
```

For the complete workflow including key generation, see [CST Workflow](04-cst-workflow.md).

## CSF Binary Inspection

To inspect a compiled CSF binary and verify its contents:

```bash
# Using NXP SPSDK's nxpimage tool (preferred):
nxpimage hab parse-hab --binary flash.bin

# Using Python script from NXP support:
python3 hab_log_parser.py --csf csf_spl.bin

# Manual hexdump inspection:
hexdump -C csf_spl.bin | head -40
# Look for:
# d4 XX XX 43  - CSF header (XX XX = length)
# c0 ...       - Install SRK command (tag 0xC0)
# bf ...       - Install CSFK / Install Key / Authenticate CSF
# ca ...       - Authenticate Data command (tag 0xCA)
```

The CSF binary size for a standard RSA-2048 signing configuration:
- Header: 4 bytes
- Install SRK + table: ~1100 bytes (4 × RSA-2048 keys)
- Install CSFK + cert: ~1000 bytes
- Authenticate CSF + signature: ~800 bytes
- Install Key + cert: ~1000 bytes
- Authenticate Data + signature: ~800 bytes
- Total: approximately 4–6 KB

Always allocate at least 8 KB for the CSF region in flash.bin.
